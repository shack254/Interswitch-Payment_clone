Product Requirements Document: Simplified Interbank Switch (PesaLink-style)

Simplified interbank switch — learning project PRD

Learning project Python / REST JSON v1.0 scope

Project scope

What we're building

A Python-based simulation of an interbank payment switch (inspired by PesaLink) that routes fund transfer requests between dummy bank instances via JSON REST APIs. Includes a reusable dummy bank model with customer records, balance management, and transaction history. The goal is to understand payment switch architecture through code — not to build production infrastructure.

Out of scope (v1): real banking integrations, cryptographic message signing, regulatory compliance (CBK, PCI-DSS), mobile/USSD interfaces, fraud detection engines, foreign currency handling, loan or credit products.

Functional requirements — switch core

  • FR-01 Accept an inbound transfer request via POST /transfer with JSON payload: sender account, receiver account, amount, currency, and transaction reference.
  • FR-02 Route the request to the correct destination bank based on the receiver's bank code (parsed from account number or a routing table).
  • FR-03 Validate the sender: confirm account exists, is active, and has sufficient balance before debiting.
  • FR-04 Debit the sender's account at the source bank and credit the receiver's account at the destination bank atomically (simulate a two-phase approach).
  • FR-05 Return a structured JSON response with a status code (SUCCESS, FAILED, PENDING), transaction ID, and timestamp.
  • FR-06 Expose a GET /transaction/{ref} endpoint to query transaction status by reference.
  • FR-07 Handle and return meaningful error codes for: insufficient funds, account not found, invalid amount, duplicate transaction reference.

Functional requirements — dummy bank model

  • FR-08 Each dummy bank instance exposes its own REST API with endpoints for account lookup, balance check, debit, and credit operations.
  • FR-09 Customer records must include: account number, full name, national ID, phone number, account status (active/frozen/closed), and current balance.
  • FR-10 Seed data must include at least 10 dummy customers across two bank instances to enable realistic test scenarios.
  • FR-11 Balance deduction must reject if the resulting balance would go below zero (no overdraft in v1).
  • FR-12 Each bank must maintain a simple transaction log (in-memory or SQLite) recording debit/credit events with timestamps.
  • FR-13 Expose a GET /account/{account_no}/balance endpoint for balance enquiry.

Non-functional requirements

  • NFR-01Testability: All switch logic and bank model logic must have pytest unit tests. Target ≥80% coverage on business logic functions (validation, routing, balance checks).
  • NFR-02Observability: All API requests and responses must be logged with method, endpoint, payload summary, status code, and latency. Use Python's built-in logging module.
  • NFR-03Error handling: No unhandled exceptions should return a 500 to the client. All error paths must return a structured JSON error body with a code and message.
  • NFR-04Portability: Must run locally with a single python main.py or uvicorn command. No external services required (no real DB, no message queue).
  • NFR-05Documentation: README must explain the system architecture, how to run the project, and include at least 3 example curl/requests calls with expected responses.
  • NFR-06Consistency: All API responses must follow a uniform JSON envelope: status, data, error, timestamp.

Suggested tech stack

FastAPI Pydantic pytest httpx (test client) SQLite / in-memory dict uvicorn python-dotenv

Switch routing logic
